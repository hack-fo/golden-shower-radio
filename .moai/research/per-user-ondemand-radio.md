# Per-User On-Demand Conversational Radio — Brainstorm

Golden Shower Radio today is one organism: a single 24/7 pull loop where Liquidsoap polls the brain every ~2s, gets one track, and feeds one Icecast mount that every listener hears in lockstep. The pivot on the table is to give each listener their *own* station — an on-demand session that only plays while they're present, remembers where they left off, greets them by context, and (eventually) lets them talk back to the host. This is a genuine architectural fork, not a feature: it breaks the project's two foundational invariants — "one shared stream" and "the stream never stops" — by introducing per-session state cardinality and a reframed golden rule ("the stream never stops *while you are listening*"). The good news buried in the research: most of the hard parts are already specified (CALLIN-003, SEEDING-029, WEBUI-018, LIKE-015), the shared infrastructure is already factored for reuse, and the conversational half is bottlenecked not by speech tech but by the one thing this project deliberately can't scale — the single Claude subscription quota.

## The core architectural fork

The decision is **replace** the 24/7 Icecast broadcast, **augment** it with a parallel per-user layer, or build a **hybrid** that keeps both. The tradeoffs:

| Path | What it does | Honest cost |
|---|---|---|
| **Replace** | Tear down the single broadcast; every listener is their own station | Discards the project's identity (roster-as-station, one shared mount), rebuilds state/playout/persona per-session, and — fatally — destroys the **single-quota amortization**. LLM cost scales linearly with listeners against an un-scalable subscription (`brain/llm.py:106` `@MX:ANCHOR`). |
| **Augment** | Keep the 24/7 broadcast as default/identity/fallback; add an opt-in per-user "personal session" layer beside it, reusing the same library/personas/TTS/knowledge | Two playout models to reason about; per-user state is new surface. But the broadcast amortizes the quota for the public station, and the personal session falls back to the broadcast on stall — golden rule preserved. |
| **Hybrid (HLS pivot)** | Replace Icecast with per-user server-side HLS streams or per-session Liquidsoap | Highest architectural coherence (true seek/resume), but HLS is timing-sensitive (manifest rewrites, segment lifecycle, CORS) and Liquidsoap becomes a large new surface. More edge cases, higher operational ownership. |

**RECOMMENDATION: AUGMENT.** The entire codebase is single-station *by construction* — one `_recent` ring, one `now_playing`, one `songs_since_talk` counter, one welcome flag (`brain/state.py:24-209`), one `/api/next` pull every Liquidsoap hits (`deploy/config/radio.liq:50`). Replacing throws away both the editorial identity (5 EN + 2 FO personas authored around one shared voice) *and* the only cost-amortization mechanism the economics depend on. Augment lets the broadcast stay the always-on default and the fallback when a personal session stalls, keeps the single quota cheap for the public station, and lets per-user grow incrementally. Crucially, **Liquidsoap is not the blocker** for this — its pull model (`request.dynamic.list` → `/api/next` polling) is exactly what *enables* per-user: change `/api/next` to accept `?user_id` and return that user's queue. The real blocker is **state cardinality**: today `StationState` is a singleton; per-user requires forking it per session or moving it to a persistent store. Keep Icecast for the broadcast (zero disruption); add per-user routes to the brain.

## Facet deep-dives

### 1. Architecture & delivery

**Key findings.** Current playout is strictly one-to-many: `radio.liq:50` runs `request.dynamic.list(id="gsr", prefetch=2, retry_delay=2.0, next_track)` calling the brain every ~2s; all listeners get one 320kbps MP3 on Icecast mount `/radio`. The brain's `/api/next` has *no* user/session context — `brain/server.py:303-335` calls `picker.pick()` against the singleton `StationState` (`brain/state.py:16-50`: one `_now_playing`, one `_recent` deque maxlen=20, one `_committed_keys`, one `_songs_since_talk`). Critically, the shared infrastructure is *already factored for reuse* (`.moai/project/codemaps/dependencies.md`): library, metadata, voice, knowledge, llm are leaf modules — acquisition, enrichment, analysis, research all read them independently. Per-user isolation is needed only at the **picker + rotation** layer.

**Options.** (A) **Hybrid, RECOMMENDED** — keep Icecast on `/radio`; add `/api/queue?user_id=...` returning a JSON track list a web player consumes, plus `/api/mark-played` to advance that user's personal rotation. Liquidsoap/Icecast untouched. (B) **Full HLS** — brain generates `/hls/USER_ID/playlist.m3u8` with per-user TS segments; true seek/resume but HLS spec complexity + Liquidsoap `input.harbor`/`output.hls` is a new surface. (C) **Client-side** — brain serves `/api/playlist` + `/stream/TRACK_ID`; all playback state in the client (localStorage); lightest server, no cross-device sync.

**Risks.** Playback-state cardinality explosion if kept in-memory (lost on restart, bloats at 1000s of users) → mitigate with a persistent `user_sessions` SQLite table from day one (~200 bytes/row). Do **not** create thousands of Icecast mounts. No auth layer exists today → start with stateless `user_id` in a cookie. Household sync: two browsers see the same `/api/queue` because the picker reads global `recent_keys` → include `device_id`/`household_id` in the session row.

### 2. On-demand lifecycle & energy

**Key findings.** A per-user session needs presence detection (browser heartbeat via WEBUI-018, or inferred from `/api/next` calls), start/stop, pause/resume with a configurable idle timeout (5–30 min), and resume-state storage. **Savings (on-demand):** the director curation loop (`brain/director.py:47-130`, 25-track `curate_batch` LLM calls), the TalkDirector TTS (`brain/talk.py`), and any conversational loop only run *while a user listens*. **No savings (shared background, always-on):** acquisition (`brain/acquire.py`, slskd/yt-dlp, shared worker pool), analysis (`brain/analyzer.py`, librosa DSP), and research (`brain/research.py`, MusicBrainz/Wikidata) — these keep the library fed for everyone and scale sublinearly. The golden rule **reframes** from CORE-001's "the station never stops" to "*a user's personal stream never goes silent while they are present; when they leave, the session legitimately ends*." Storage rides DATASTORE-022's four-file partition — session rows go in `state.db` with the same WAL/RLock discipline; session events append to `events.db`. No Redis, no new engine.

**Resume-state schema.** `{session_id (UUID), user_id, last_track_key, position_seconds, last_resumed_at, time_of_day_resumed, inferred_mood, recent_played[], session_started_at, session_last_active_at}`.

**Options.** Additive HTTP session API (`/api/session/start|resume|pause|status` + per-user `/api/next`) over an unchanged broadcast (minimal, non-breaking) · refactor the director loop to per-user "session queue" (cleaner, more invasive) · defer to a single "bookmarked position" prototype for the admin user · feature-flag both behind `BRAIN_PER_USER_SESSIONS` for gradual rollout.

**Risks.** `state.db` WAL is one-writer-at-a-time — 100+ concurrent heartbeat updates could bottleneck → batch heartbeats (1 update / 10s / user). Resume-after-network-loss correctness needs explicit grace periods. Idle timeout interacts with talk cadence and `recent_keys` — does the counter reset or carry across a multi-hour gap? Decide early.

### 3. Identity, state & memory

**Key findings.** Single-tenant is the current reality — rotation is global, `library.mark_played()` updates one shared `library.json` for all. Identity should be **lightweight and optional**: a 64-char device UUID in a cookie/localStorage, generated on first `/api/next`, used directly as the state key (no session table, no login). Reuse LIKE-015's signed HMAC token pattern (secret in gitignored `secrets/brain.env`) to prevent forgery. The per-user profile schema is *mostly pre-designed*: SEEDING-029 taste config (ANCHOR/COMPASS/WOPR + optional `seed_reference`), LIKE-015 per-recording affinity, STATS-013 `play_events`. The clean split is **global library = SOURCE, per-user state = CONSUMPTION**: what's per-user is the `recent_keys` ring, `last_committed_path`, taste seed, welcome flag, and resume position. State generalization is mechanical — keep `StationState` for global truth (Icecast now_playing, website HTML), add a sibling `dict[user_id → UserState]` for per-user rings/cadence/welcome (`brain/state.py:16-50` → `brain/server.py:303-335` routes `picker.pick(user_id)`). Taste seeding is a populated stub: `director._seed_reference()` returns `[]` today (`brain/director.py:47-51`); `curate_batch(seed_reference=...)` already accepts it as non-binding bias (`brain/llm.py:211-236`).

**Privacy (LIKE-015 REQ-LP-002): aggregate-only, never per-listener surveillance.** Store taste seed + like/drop event IDs + optional position/mood; cap listening history to a 100-track ring; 90-day rolling `play_events`; user-initiated delete endpoint. Likes/drop-off must key on **canonical track identity** (ENRICH-012 spine; degrade to `Track.key` dedup slug) so affinity isn't fragmented across duplicate copies.

**Options.** Device-token only (zero friction, no cross-device) · optional OAuth + device-token fallback (durable, sync, honors deletes) · hybrid storage: **UserProfile as JSON** (atomic-write like `library.py:216-288`), **`play_events` in SQLite** (STATS-013 owns it, add `user_id` FK) — recommended.

**Risks.** Shared-library rotation races if many users hit `mark_played()` concurrently → lock per-user state separately; **don't** make `play_count` per-user for MVP. Welcome/talk explosion if 100 users connect at once → reuse one global welcome on first broadcast, per-user only after reconnect. Seed never used if the director curates globally → for MVP make the seed a *soft pick-time bias*, not a per-user re-curation.

### 4. Contextual greetings & news

**Key findings.** The host-talk machinery is load-bearing and extensible — `brain/talk.py:126-194` already builds welcome and next-song context via `generate_talk_script → produce_talk_clip`. Time awareness exists (OPS-004 REQ-OA-009 mandates Atlantic/Faroe tz; KNOWLEDGE-008 REQ-KF-002 uses the ORCH-005 world model), but **per-user** greetings need the *user's* timezone (from the client: `Intl.DateTimeFormat().resolvedOptions().timeZone`) and their last-seen time. Morning news has a real grounding path: OPS-004 Group OG (autonomous sourcing from trusted feeds) + KNOWLEDGE-008 Group KI (dated, sourced, consensus-gated facts, multi-source firewall REQ-KS-006) feed the same TTS/LLM renderer; default to **music news** (artist releases, label news) which is on-brand and simpler than general news. Anti-hallucination is already built — talk context comes from server state (the airing log), not LLM free-recall, so it's verifiable. The 8GB GPU adds no load here: greetings are CPU text + remote Claude API.

**Options.** v1 broadcast-time greeting (one "good morning at 7am Faroe time" for all — no per-user state) · v2 per-user "welcome back, you were listening yesterday at 2pm" overlay · **hybrid (recommended): v1 broadcast greetings, v2 per-user**, the pipeline supports both unchanged · music news from KNOWLEDGE-008 grounding (straightforward) · taste-filtered personalized news (v2, deferred) · context-aware tone via existing context dict + HOST_PERSONA prompt (low effort).

**Risks.** Timezone acquisition is client-dependent (fallback to station tz, correct ~90%). Morning news needs robust freshness gating (pre-schedule research at 6am, pre-render, fallback to "this week's new releases"). Collision between an auto-greeting and the user's `/api/next` pull → TalkDirector checks "is a greeting already pending?" Long-gap returners (6 months) may warrant a different script (configurable threshold, >7 days = special greeting).

### 5. Conversational host tech

This is the most-misjudged facet, so the split must be explicit.

**Key findings.** A two-way voice conversation is genuinely feasible on this box, and it's *already specified* as **CALLIN-003 Tier 2**: browser WebRTC (Pipecat `SmallWebRTCTransport`, P2P/serverless) → faster-whisper STT → LLM turn → Kokoro TTS → back, with VAD/barge-in. **STT and TTS are NOT the bottleneck:** faster-whisper large-v3 INT8 ~2.5–3GB (no meaningful WER loss); Kokoro-82M hits RTF ~0.03 (≈33× real-time), sub-100ms first chunk, ~680ms query-to-first-word — STT(~2.5GB) + Kokoro(~2–3GB) ≈ 5GB on the 8GB Ada with headroom (sources: localaimaster.com faster-whisper guide; gigagpu.com Kokoro-vs-XTTS latency benchmarks). The bottleneck is the **LLM turn**: `brain/llm.py:131-136` is deliberately blocking/non-streaming (`"".join(chunks)`, `max_turns=1`, `ANTHROPIC_API_KEY` stripped for subscription OAuth — verified at `brain/llm.py:100-119`), giving 1–10s turns that make natural conversation impossible as-is.

**The one new load-bearing finding vs the spec:** CALLIN-003's open question R-C-1 ("does the bundled claude SDK expose consumable streaming events, or must Tier 2 use a separate pay-per-use key?" — marked UNVERIFIED) is **now answered by official docs**: `claude-agent-sdk` supports `include_partial_messages=True`, yielding `StreamEvent → content_block_delta → text_delta` token deltas — exactly the first-clause-streaming primitive Tier 2 needs (source: https://code.claude.com/docs/en/agent-sdk/streaming-output). It *may* work on the existing subscription/OAuth path; **one runtime check remains** (does the bundled CLI honor it under OAuth, and does streaming a clause still count as one turn against the 5h quota?).

**Why cascaded, not hosted S2S:** native speech-to-speech (OpenAI Realtime, Nova Sonic, Gemini Live) wins raw latency (TTFT ~0.78–2.98s) but is hosted (cost + privacy) and discards the transcript — which this station *needs* for the fail-closed moderation floor + classifier + append-only audit log + per-track grounding (sources: softcery.com real-time-vs-turn-based; inworld.ai speech-to-speech). A proven open reference (Modal: Parakeet + Qwen3-4B-vLLM + Kokoro + Silero VAD over WebRTC ≈1s voice-to-voice; modal.com/blog/low-latency-voice-bot) validates v2 is achievable — but its sub-second number relies on a *local* small LLM with tuned TTFT; Claude-over-network will be higher and jittery.

- **v1-achievable (works today on the blocking path, gated only on GPU-in-Docker):** push-to-talk / async voice notes. Listener holds a button, speaks; faster-whisper transcribes; the existing blocking persona turn (+ grounding) replies; Kokoro renders the persona voice; play it back. No real-time turn-taking, no barge-in — lag is fine because the UX is explicitly async. This is CALLIN-003 "Tier 1" reframed for the per-user product.
- **v2-aspirational (~1–1.5s perceived onset):** streaming turn-taking. Wire a new streaming path with `include_partial_messages=True`, pipe the **first clause** straight into Kokoro while the rest generates, add Silero VAD + turn detection (SmartTurn) + barge-in (Pipecat interruption frame) over WebRTC. Honest expectation: "usually good, occasionally laggy," not guaranteed sub-second.

**Per-persona voice reconciles cleanly:** VOICE-002 already provides a 10-voice Kokoro palette (1:1 voice↔persona, `brain/voice.py:218-223`); the loop renders the active persona's voice, the persona prompt + KNOWLEDGE-008 grounding feed the turn (`brain/llm.py:261-385`). The per-user/on-demand angle *helps*: a conversation is inherently 1:1, so a single GPU serving one active speaker is the natural concurrency bound, sidestepping the multi-tenant 8GB problem.

**Risks.** **GPU-in-Docker is the critical-path prerequisite** — `Dockerfile.brain:47,60` installs CPU-only torch; on CPU even v1 is sluggish and starves the music brain (owned by ANALYSIS-006 / GPU-enablement, not this facet). The subscription-OAuth streaming question is verified in docs but **not on this exact stack** — runtime-confirm OAuth passthrough + quota accounting; if it fails, v2 needs a separate pay-per-use Haiku key (recurring cost). Pipecat `SmallWebRTCTransport` has a flagged "robotic/choppy audio" regression from v0.0.62 — pin and spike a known-good version. CALLIN-003's heavyweight broadcast-delay/dump/harbor-takeover rails protect a *shared public air feed*; a **private 1:1 conversation has a far smaller liability surface** — don't over-build v1 by pulling them in wholesale.

### 6. Product shape, cost & scaling

**Key findings.** **AUGMENT, not REPLACE** (see fork above). The **LLM subscription quota is the load-bearing scaling wall** and does *not* scale per-user: every curation/talk call shells `claude-agent-sdk` → bundled CLI with `ANTHROPIC_API_KEY` stripped against one `~/.claude` OAuth subscription's 5-hour rolling quota (`brain/llm.py:100-119`; `@MX:ANCHOR` at `:106` warns "every LLM call ships this; burns the 5h quota"). One shared broadcast consumes a fixed slow drip (one `curate_batch` per ~25 tracks, `brain/config.py:63`; one talk script per cadence) *amortized across all listeners*. Per-user multiplies that drip by concurrent sessions against one quota — **there is no API-key path to scale out** without breaking the subscription contract. **Where it breaks:** a few concurrent *conversational* sessions saturate both the quota *and* the 8GB GPU (Whisper-large + dialogue LLM + Kokoro won't co-reside for even one heavy session). Practical ceiling on this box: single-digit concurrent music-only sessions; conversational is effectively ~1-at-a-time.

**Per-persona taste survives by ASSIGNMENT, not duplication:** a personal session *picks or is assigned* one of the 7 existing personas and inherits that persona's taste, with the per-user seed as an *additional light bias* on top (SEEDING-029 non-binding `seed_reference`). Building N new per-user taste models would both homogenize taste (violating the no-convergence philosophy) *and* multiply LLM cost — the worst of both axes.

**Comparables validate the shape.** Subwave (github.com/perminder-klair/subwave) is an architectural twin — Icecast single mount, LLM picks from your own library, talks between tracks, Piper/Kokoro, one Linux box — and it *also* chose shared-broadcast, confirming augment is the proven shape and that per-user on-demand on one box is genuinely unclaimed. Spotify's AI DJ added voice in 2025 but it is **push-to-talk / request-shaped**, not free conversation (newsroom.spotify.com 2025-05-13; techcrunch.com 2025-10-15) — Spotify itself frames natural two-way dialogue as a *future* plan. So free-form voice is frontier even at Spotify scale; scope it as a later, capacity-capped feature.

**Risks.** Subscription-quota saturation is the dominant failure mode — any per-user design needs a **per-session LLM budget + hard fallback to the shared broadcast on exhaustion**. On-demand "only generate when listening" must fall back to the always-on broadcast, never to dead air. Per-session persistence (last position, recent ring, last-seen) doesn't exist today and must stay *off* the <1s pull path. No horizontal scaling path exists — set an explicit concurrency cap and queue/fallback rather than pretend to scale.

## Energy/cost reality

**The on-demand win is real and clean.** When a listener is away, their personal session spends *nothing*: no curation LLM call, no TTS render, no conversation loop. Only the shared background work (acquisition, analysis, research) keeps running — and that runs 24/7 anyway to feed the library for everyone, so it's not a new cost. So a personal session's marginal cost is zero while idle and only the on-demand TTS/curation/conversation while active. This is the genuine efficiency story versus an always-on per-user broadcast (which would burn quota and GPU around the clock per user).

**Where it breaks at scale — the single box + single subscription:**
1. **The 5-hour OAuth quota is shared and un-scalable.** One broadcast amortizes it across all listeners; N personal sessions create N independent curation+talk streams against the *same* quota (`brain/llm.py:106`). A handful of simultaneous sessions — especially conversational, which makes the drip *interactive* rather than slow-paced — exhausts it, starving both the personal sessions *and* the public broadcast. There is no per-seat billing path that respects the subscription contract.
2. **The 8GB GPU is a hard single-conversation ceiling.** STT(~2.5GB) + Kokoro(~2–3GB) ≈ 5GB fits *one* active conversation with headroom; a second concurrent conversation, or a background analysis/Whisper job competing for the Ada, risks OOM. Needs a VRAM admission/queue guard.
3. **No horizontal path.** One box, one GPU, one subscription. The product must set an explicit concurrency cap (single digits for music-only; ~1 for conversational) and **queue / "host is busy" / transparently drop to the shared broadcast** past it — not pretend to scale.

The mitigations are all the same shape: per-session LLM budget, VRAM admission guard, and a hard fallback to the always-on broadcast.

## Recommended phased direction + SPEC decomposition

**Smallest MVP that proves the value — "resume + greet":** a personal session keyed by a device UUID, reusing the *shared* library and *one assigned* persona, that remembers your last position + recent ring + last-seen, and opens with a context-aware greeting via the already-built persona TTS. **No per-user curation LLM, no STT, no conversation.** Concretely: (1) session-id-keyed state wrapping `StationState` (`brain/state.py`) — per-session recent ring + last position + last-seen, instead of the single global; (2) `/api/next?session=<id>` pulling that user's next track from the shared library (`library.pick_eligible(avoid=...)` already supports per-listener avoid sets); (3) on (re)connect, fire the existing welcome/talk path (`state.arm_welcome`, `talk.TalkDirector`) with time-of-day + last-session context; (4) on-demand = the personal source spins only while connected, falling back to the 24/7 broadcast on stall (golden rule preserved). This is *incremental* — it parameterizes state + the endpoint rather than replacing them.
