# AUDIT — SPEC-RADIO-IMAGING-010 (Imaging Production: local-gen / hosted-break / Suno fallback)

Auditor: plan-auditor (adversarial, independent). Reasoning context from the SPEC author was
ignored per M1 Context Isolation. Audit is based only on `spec.md` + `acceptance.md` as written.

## Verdict: SHIP-WITH-FIXES

Structurally sound: REQ/AC parity is mechanically clean (38 REQ + 10 NFR = 48 spec items, exactly
48 acceptance entries, all IDs 1:1). Version/changelog matches content. EARS compliance is strong.
BUT the v0.3 "local-gen becomes PRIMARY/fully-autonomous + on-GPU sidecar in-scope" expansion
updated the spec body and added Group IG/IH ACs + Scenarios 8/9 — yet left two Section-A NFR
acceptance entries (and three glossary rows + Section 1.5) carrying the pre-v0.3 framing. The result
is two hard internal contradictions where an acceptance criterion now FAILS the very requirement it
traces to. These are localized text-sync defects, not structural failures — hence SHIP-WITH-FIXES.

## Must-fixes (with REQ/AC IDs + concrete fix)

### MF-1 — AC-NFR-I-5 contradicts its own requirement NFR-I-5 and all of Group IG
`acceptance.md` AC-NFR-I-5 reads: *"VERIFY no requirement/AC/doc/website-copy claims 'fully
autonomous generation'; the stated scope is human-seeded generation + fully-autonomous
post-production."* This is the pre-v0.3 (v0.2) text. The spec's own NFR-I-5 (spec L1420-1428) now
**requires** the opposite: *"The PRIMARY local-generation path (Group IG) IS genuinely
fully-autonomous… describing it so is HONEST and required."* AC-IG-007 and Scenario 8 explicitly
claim local-gen is "fully autonomous." A tester following AC-NFR-I-5 literally would FAIL the SPEC
for satisfying NFR-I-5.
**Fix:** Rewrite AC-NFR-I-5 to scope the prohibition to *Suno* only — e.g. "VERIFY the LOCAL path
(Group IG) is described as fully autonomous (honest, required); VERIFY no requirement/AC/doc/website
copy claims fully-autonomous generation via *Suno*; the Suno seed is human-seeded and Group IX is a
labeled ToS-violating opt-in POC; no third-party music-generation API/wrapper is called."

### MF-2 — AC-NFR-I-7 asserts "no on-GPU generative model" + "no new service", contradicting Group IG
`acceptance.md` AC-NFR-I-7 reads: *"VERIFY the implementation adds … no new service, no on-GPU
generative model, and no change to the primary `%mp3` mount."* v0.3 explicitly PROMOTED the on-GPU
generative model into scope (REQ-IG-001/004; Section 14 "PROMOTED IN v0.3 — no longer a future
fork") and the spec's NFR-I-7 (spec L1439-1448) now sanctions *"the one … additive external GPU
service."* AC-NFR-I-7 directly contradicts REQ-IG-001/004 and its own requirement.
**Fix:** Replace "no new service, no on-GPU generative model" with the v0.3 language: "the local-gen
`music-gen.server` sidecar is the ONE sanctioned additive external GPU service (the brain POSTs to it
like Liquidsoap/TTS); no NEW brain-internal service; no second serving path; no new datastore; primary
`%mp3(bitrate=320)` mount unchanged."

### MF-3 — Glossary (Section 3) + Section 1.5 still frame human-seeded Suno as the ONLY/primary bed source
- Glossary "**Bed**" (spec L531): "generated (by a human, in the Suno UI)" — omits the now-PRIMARY
  local ACE-Step source.
- Glossary "**Human-seeded generation**" (L532): "The ONLY non-autonomous touchpoint" — contradicts
  NFR-I-5/Group IG (local path has NO human touchpoint and is the default).
- Glossary "**Drop dir / ingest dir**" (L533): "The watched directory the human replenishes" —
  contradicts REQ-IG-005 (the sidecar writes raw clips into the same dir).
- Section 1.5 (L378-379): "The single human touchpoint (occasional bed generation in the Suno UI)…"
  — same stale "single human touchpoint" framing.
**Fix:** Update all four to be source-agnostic: the bed is generated by the local ACE-Step sidecar
(primary/default) OR human-seeded in the Suno UI (optional premium); the drop dir is replenished by
the local-gen sidecar (primary) or a human (optional); on the local path there is NO human touchpoint.

## EARS findings

EARS compliance is strong — every requirement opens with a recognizable EARS clause; no free-form
prose requirements, and no Given/When/Then test scenarios mislabeled as requirements in the spec
bodies. Concrete issues, all minor:

- **REQ-IS-004** declared "Unwanted" in the traceability table (spec L1639) but written as
  "The system SHALL NOT return a malformed or multi-track item…" — that is a Ubiquitous prohibition,
  not the Unwanted "If [condition], then the system shall …" structure. Type-label mismatch.
- **REQ-IG-003** declared "Unwanted" (spec L1647) but written "The system SHALL NOT use MusicGen,
  AudioGen…" — same Ubiquitous-prohibition-vs-Unwanted-label mismatch.
  (Fix either: relabel both as Ubiquitous, or rephrase to "If an engine carries NonCommercial weights,
  then the system shall NOT use it on air.")
- **REQ-IX-002 / REQ-IX-003** open "Where … the system MAY drive / MAY navigate …" — MAY-only,
  permissive, no binding `shall` on the core behavior (the normative weight sits in later
  "[HARD] … is the rail" clauses). Weak normativity. Acceptable given the group is explicitly
  EXPERIMENTAL/OPT-IN, but worth tightening the binding clause. (REQ-IX-001/004 correctly pair
  "Where…" + a binding SHALL.)

No EARS *violations* that block. ears_violations list below = the 2 type-label mismatches.

## Boundary / single-source-of-truth notes

- **OPS-004 (REQ-OE-010 ledger/gate):** cleanly EXTENDED with additive classes (REQ-IB-004 paid-tier,
  REQ-IG-005 local-gen self-generated). Gate mechanism + store not re-owned. OE-002/005/006/007/009/
  011/012 fulfilled by number, not restated. Correct EXTEND behavior, no duplication.
- **ANALYSIS-006:** loudness metering reused, not re-owned (REQ-IP-005). OK. **Minor boundary note:**
  REQ-IP-002 "phrase-snap to a musical phrase boundary" is treated as a self-contained TUNABLE
  heuristic but musical-phrase/beat detection plausibly belongs to ANALYSIS-006 (cues/beats). The SPEC
  does not reference ANALYSIS-006 for the phrase-boundary signal — either it should consume an
  ANALYSIS-006 cue, or explicitly state phrase-snap uses a local non-feature heuristic.
- **CORE-001 (radio.liq):** REQ-IS-002 adds a no-hard-cut IDENT transition in radio.liq. The SPEC
  frames it as "coordinates with CORE-001 … referenced, not re-owned" and leaves the `%mp3(320)` mount
  unchanged — but a new ident transition path IS a radio.liq (CORE-001-owned) edit. Acceptable as a
  coordinated extension, but the ownership seam should be stated more sharply (who lands the radio.liq
  change). Boundary tension, not a REQ duplication.
- **TAGSTREAM-009 / KNOWLEDGE-008:** no overlap. Imaging clip catalog (REQ-IP-008) is the OPS-004
  CLIPS_DIR catalog, distinct from TAGSTREAM music-file tag/artwork writes. Hosted-break scripts come
  via PROGRAMMING-007 (which owns the KNOWLEDGE-008 grounding), not directly. Clean.
- **DATA-vs-CODE rail:** RESPECTED. AI self-expansion targets are the jingle-spec TABLE (YAML/JSON
  data, REQ-IL-001 "the AI may evolve it"), per-show palette presets (REQ-IL-003), and caption presets
  (REQ-IG-001) — all config/data stores. `config.py` targets and `radio.liq` mount are explicitly
  "reused/unchanged." No requirement authorizes the AI to edit source during normal ops.
  **Nit:** this rail is implied but never stated as an explicit requirement — consider an explicit
  "autonomous evolution writes config/data only, never source" guard for robustness.

## Over-engineering / feasibility

- **Group IX (5 REQ) is near-redundant gold-plating now that Group IG exists.** Once local ACE-Step is
  the PRIMARY fully-autonomous path, the marginal value of a ToS-violating, captcha-fragile,
  ban-risk headed-browser Suno automation is near-zero. It is honestly gated (opt-in/off-by-default/
  POC, NFR-I-8), so it does not endanger anything — but carrying it as 5 in-scope REQ + a Playwright/
  vision/NopeCHA surface is speculative weight beneath a path that already solves the autonomy goal.
  Recommend explicitly marking IX "not built unless local-gen proves insufficient."
- **Group IG status-vs-readiness mismatch (feasibility):** IG is declared PRIMARY / DEFAULT-ON, but
  REQ-IG-004's nvidia-container-toolkit + CUDA-torch passthrough is "being set up" (not plumbed), and
  R-I-14 leaves the WSL2 Docker CUDA-torch + Python≥3.12 + vLLM enforce_eager question OPEN. The SPEC
  honestly provides a CPU-fallback (REQ-IG-006), so it is feasible — but "default-ON PRIMARY" overstates
  build-time reality; the realistic initial default is the CPU-fallback batch until the GPU sidecar is
  plumbed. Consider softening "default-ON" to "default once the GPU sidecar is plumbed; CPU-fallback
  batch until then."
- Production pipeline (Group IP) is correctly an extension of shipping ffmpeg/pyloudnorm/Kokoro code
  (`voice.py:340-342` seam) — feasible, no new deps. Good.

## Nits

- Section 1.2 bullet headers carry leftover dossier shorthand "(refuted)" / "(refuted as 'clean')"
  which read confusingly in a requirements doc; consider plain phrasing.
- "Total: … 1:1 REQ↔AC" is asserted in three places (HISTORY v0.3, Section 15, acceptance Section C) —
  fine, but keep them in sync if counts ever change (single-source-of-truth risk).
- REQ-IG-005 phrasing "When a local-generation batch runs, the system SHALL run it as a … BATCH" is
  mildly circular; trigger could be "When a local-generation batch is scheduled."
