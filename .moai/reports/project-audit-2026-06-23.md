## Why SPECs aren't on GitHub + board recommendation

Every spec's `issue_number: null` and no issues/Project were ever created — the `.moai` workflow uses local `.moai/specs/` files as the single source of truth, with `issue_number` reserved as the link slot a GitHub sync step *would* fill. Durable to the repo, but invisible as queryable work items.

**Bespoke Projects v2 board (org `hack-fo`):** fields = **Status** (single-select, the taxonomy above), **Group** (9 values), **Version** (text), **Number** (1–28), optional **Impl** (yes/partial/no, to diff intent vs measured code). One item per SPEC. Represent `dependsOn` via a **"Depends on" text field** (handles cycles, mirrors frontmatter 1:1) rather than task-list DAG edges. Do NOT create nodes for `IMAGING-005`/`OPS-001` (non-existent).
