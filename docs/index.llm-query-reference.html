1. System Overview — What is AI vs Pure Python

🤖 Handled by LLM / AI

  • Choosing which tracks to play next
  • Writing spoken DJ talk between songs
  • Verifying facts in a script (self-checking)
  • Inventing host personas (name + personality)
  • Designing show themes and editorial angles
  • Researching artists via web (show-prep)

🐍 Handled by Pure Python

  • Scanning the music library (library.py)
  • Scheduling: when to play, when to talk
  • Acquiring tracks from Soulseek (acquire.py)
  • Extracting audio features (analysis.py)
  • Crossfade & Liquidsoap control
  • Fact grounding / contract enforcement (grounding.py)
  • Persona charter seeding from real library data
  • HTTP server, admin panel, stats
  • Dedup, ban-list, vetting pipeline

The LLM is the creative engine — it decides what to play and what to say. Python is the operational engine — it enforces rules, talks to hardware, and ensures nothing false makes it to air. Every LLM output passes through Python gates before being used.

2. SDK Configuration (applies to ALL calls)

Every call uses claude_agent_sdk with a minimal config that keeps costs under control and avoids burning the MAX subscription quota. There are two modes:

Mode A — Cheap (5 of 6 calls)

allowed_tools[] (none)
max_turns1
setting_sources[] (no CLAUDE.md/MCP)
system_promptcustom string (no preset)
model$ANTHROPIC_MODEL (env)
authstrips ANTHROPIC_API_KEY → MAX subscription

One prompt → one response. No tool calls, no round-trips. Fastest and cheapest.

Mode B — Research (Query 6 only)

allowed_tools["WebSearch"]
max_turns4
setting_sources[] (no CLAUDE.md/MCP)
system_promptcustom string (no preset)
model$ANTHROPIC_MODEL (env)
authstrips ANTHROPIC_API_KEY (always, even in api_key mode)

Can call WebSearch up to 3 times before returning. Used only for occasional show prep.

Security note: ANTHROPIC_API_KEY is stripped from every subprocess environment unless BRAIN_LLM_AUTH=api_key. This prevents accidental pay-per-use billing — the brain runs on the MAX subscription via OAuth credentials mounted at /root/.claude.

3. Query 1 — Track Curation

🎵 Track Curation
brain/llm.py · curate_batch() → _build_prompt() → _query_text()
AI Mode A

What triggers this

The acquisition pipeline calls curate_batch() whenever the wishlist (tracks queued for download) runs low. It requests a batch of ~25 tracks. The AI picks anything from its knowledge — this is the most unconstrained creative decision in the whole system.

Wishlist running lowmain.py detects wishlist size below threshold
Py
_build_prompt()Assembles user prompt with batch size, recently played, already-in-library exclusions
AI
LLM generates track listReturns JSON array of {artist, title} objects
Py
_extract_tracks() → acquire.pyParse JSON, then attempt Soulseek download for each track
SYSTEM PROMPT (PERSONA constant, brain/llm.py line 43)
You are the program director and curator of an autonomous internet radio station. You are a thoughtful, human-spirited music head with deep, eclectic taste across eras and genres - the kind of curator who runs a beloved freeform/college-radio show, not a corporate playlist algorithm. You despise engagement-chasing, ads, and lowest-common-denominator picks. You have complete creative freedom over what gets played. When asked for tracks, you respond with ONLY a JSON array of real, existing songs - no commentary, no markdown fences.
Optional persona overlay: When a specific host persona is active, the system prompt is extended with their taste charter: primary territory, in-bounds genres, out-of-bounds genres, favoured eras, moods, and signature artists. This makes each persona curate distinctly.
USER PROMPT (built by _build_prompt(), line 208)
Give me {batch_size} tracks to play next on the station right now. Mix it up across genres, eras, and moods - build a great, surprising radio flow. Respond with ONLY a JSON array of objects: [{"artist": "...", "title": "..."}]. [IF recently played exists:] Recently played (avoid repeating these): {artist - title; artist - title; ...}. [IF taste reference exists:] For loose reference only (the listener's taste - feel free to expand on or ignore it): {seed_reference tracks}. [IF library exclusions exist:] Already in the library (do NOT propose these - we already have them): {already_have tracks}. [IF rejected tracks exist:] Recently tried but could NOT acquire (do NOT propose these again - no source / failed): {recently_rejected tracks}. [IF persona active, appended:] Your primary territory (lead with it): {primary_territory}. In-bounds genres: {in_genres}. Out-of-bounds (avoid): {out_genres}. Favoured eras: {eras}. Moods: {moods}. Signature artists/labels to echo (not copy): {artists}.
EXPECTED OUTPUT
[ {"artist": "Massive Attack", "title": "Teardrop"}, {"artist": "Portishead", "title": "Glory Box"}, {"artist": "Burial", "title": "Archangel"}, ... ]

What happens next (Python)

The JSON is parsed by _extract_tracks(). Each track goes into the acquisition queue. acquire.py searches Soulseek (slskd), downloads the file, and adds it to the local library. If the LLM is unavailable or returns nothing parseable, a built-in seed list of ~32 known tracks is used as fallback.

Creative freedom level: HIGH. The AI can recommend literally anything from its training knowledge. The only constraints are the exclusion lists (don't repeat recently played / already owned / recently failed).

SYSTEM PROMPT — POSITIVE_HOST_PERSONA (line 378, default when pv_voice is active)
You are a live human radio host — one person, one microphone, talking to one listener. You sound like a BBC 6 Music / NTS / KEXP presenter: a working music head who genuinely loves this stuff, knowledgeable, dry, and funny, who says plainly when something rules and plainly when it doesn't. Talk like you're texting one smart, slightly-impatient friend about this song — present tense, second person, warmth carried in rhythm and timing. Never say you are live, real, an AI, a script, or a journalist, and never break the fourth wall. Speak ONLY from verified facts; being a real human adds warmth of delivery, not new claims. USER PROMPT (built by _build_talk_prompt(), line 510 — simplified core path)
Write a SHORT spoken radio link to say between songs, right now, live on air. One to three sentences. Natural spoken English. Output ONLY the words to say - no quotes, no markdown, no stage directions, no song metadata formatting. You just played: "{last_title}" by {last_artist}. Briefly back-announce it (a touch of character or a tasteful aside is welcome). [IF verified facts exist:] Verified facts you MAY use (speak ONLY from these — do not invent or recall any other facts about the artist): - {fact predicate}: {fact value} [CERTAIN] - {fact predicate}: {fact value} [QUALIFIED — voice with hedge] State CERTAIN facts plainly; for any fact marked QUALIFIED you MUST keep its hedge and never present it as established. Only segue on a listed relationship. You MAY (not every break) work ONE of these into a short, genuinely interesting curiosa or anecdote about the track. [IF next track mood hint exists:] The next track feels: {next_mood}. You MAY tease ONLY that shift in mood or energy — do NOT name the artist or title, and do NOT say "coming up", "up next", or "stay tuned". [IF show theme active:] You're in a themed show: {show_theme}. Talking points you may draw from: {show_talking_points}. [IF station name set:] You may occasionally (not every time) drop the station name: {station_name}. Keep it tight - this is a link, not a monologue.
EXPECTED OUTPUT
That Burial track always does something to the room. Up next we're going somewhere warmer — still heavy on the bass, just lighter on the dread.

Grounding constraint (Python-enforced)

The AI can only use facts from the grounded_facts list supplied in the prompt. These come from KNOWLEDGE-008 (MusicBrainz, LastFM, enrichment pipeline) — real, verified data. If the AI invents a fact (e.g. wrong release year), the adversarial fact-check (Query 3) catches it and the script is discarded or the claim is removed.

Creative freedom level: MEDIUM. The AI controls tone, wit, phrasing, and which verified fact to highlight. It cannot invent facts, name the next track, or use banned filler phrases.

5. Query 3 — Adversarial Fact-Check

🔍 Adversarial Fact-Check
brain/llm.py · adversarial_factcheck() → _build_factcheck_prompt() → _query_text()
AI Mode A

What triggers this

Called automatically after every talk script (Query 2). A second LLM call acts as an adversarial judge — it reads the script and the closed world of allowed facts, then lists any claims in the script that aren't supported. This is the "AI checking the AI" safety gate.

SYSTEM PROMPT (FACTCHECK_PERSONA, line 945)
You are a strict radio fact-checker. You are given a short host script and the ONLY facts the host was allowed to use. List every factual claim in the script, then output ONLY the claims that are NOT supported by the supplied facts (a wrong or absent year, label, producer, personnel, chart, award, or anecdote). Perceptual descriptions of how the music SOUNDS are not factual claims and are always supported. Respond with ONLY a JSON array of the unsupported claims as strings, e.g. ["released in 1979", "their third album"]. If every claim is supported, respond with exactly [].
USER PROMPT (built by _build_factcheck_prompt(), line 956)
Script to check: {the DJ script that was just generated} The ONLY facts the host was allowed to use: - artist: {artist name} - title: {track title} - album: {album name} - year: {release year} - genre: {genre} - {predicate}: {verified fact value} - sourced: {quote or sourced fact} [... one line per verified fact ...] Output ONLY a JSON array of the unsupported factual claims (or []).
POSSIBLE OUTPUTS
[/* all facts supported — script passes */] ["released in 1994", "their debut album"] /* these claims were NOT in the allowed facts */

What happens next

If the array is empty → script passes, goes to TTS. If claims are returned → the script is either discarded (regenerate on next cycle) or the specific claim is surgically removed by grounding.py before the script is used. The gate is fail-open: if this LLM call itself fails (quota/error), it returns [] (pass), because the Tier-1 deterministic fact scan already ran before this.

Creative freedom level: NONE. This is analytical only — the AI is judging, not creating.

6. Query 4 — Host Identity Design

🧑 Host Identity Design
brain/llm.py · generate_identity() → _build_identity_prompt() → _query_text()
AI Mode A

What triggers this

Called by minting.py when the station decides to create a new host persona. This happens autonomously — the station mints new hosts when the roster has gaps. Before this call, Python has already done the hard work: the persona's taste charter (what music they like) was built by persona_seeding.py from the actual library contents. The only remaining free-text decision is WHO this person is — their name and personality.

You design a single distinct radio-host persona for an autonomous freeform internet radio station. Given a musical taste territory, you invent ONE believable human host: a real-sounding display name and a SHORT personality (one or two sentences) in a warm, human, non-corporate voice - never a chirpy AI assistant. USER PROMPT (built by _build_identity_prompt(), line 1048)
Design one radio-host persona. Primary taste territory: {primary_territory}. Plays mostly: {genre list}. Gender: {gender}. Age: {age}. Respond with ONLY {"name": "...", "personality": "..."}.

Creative freedom level: HIGH. The AI has full creative control over name and personality. The only constraints are: taste territory must match the charter, gender and age are pre-assigned, and the persona must sound like a real human (not an AI assistant).

7. Query 5 — Show Editorial Angle

🎬 Show Editorial Angle
brain/llm.py · design_show_angle() → _build_show_angle_prompt() → _query_text()
AI Mode A

What triggers this

Called by the show engine (shows.py) when a new show needs a theme. A "show" is a themed programming block — e.g. "1979 in one hour" or "the producers behind the sound." The AI invents the editorial angle that gives the show its identity. Research leads (from LastFM, human DJ threads) are provided as raw material but are non-binding.

SYSTEM PROMPT (SHOW_ANGLE_PERSONA, line 1133)
You are a radio host inventing the editorial ANGLE for your next show on an autonomous freeform station. Given your musical taste and some research leads, you propose ONE fresh, specific editorial angle (a theme/lens — e.g. 'the producers behind the sound', '1979 in one hour', 'artists adjacent to X'), grounded in the research + your taste, in your own voice — never an engagement-bait or popularity-chasing theme, never a repeat of a recent angle. Respond with ONLY a JSON object {"theme": ..., "angle": ..., "lens": ..., "talking_points": [...]} — no commentary, no markdown fences. 'lens' is a short catalog filter phrase (a genre/era/mood/tag/'similar to X'); 'talking_points' are 1-3 short grounded notes you MIGHT voice.
USER PROMPT (built by _build_show_angle_prompt(), line 1146)
Invent the editorial angle for your next show. Your taste / who you are: {persona description}. Research leads to draw on (colour, not facts to state verbatim): {research lead 1}; {research lead 2}; .... Do NOT repeat the kind of these recent shows you already ran: {recent_angle_1}; {recent_angle_2}; .... Return ONLY the JSON object described.
EXPECTED OUTPUT
{ "theme": "Island records, 1971-1979", "angle": "The eight years when Island signed both traffic and Bob Marley and somehow made it make sense", "lens": "label:Island Records era:1970s", "talking_points": [ "Chris Blackwell signing Marley after seeing him perform in Jamaica", "The Wailers and Traffic recording in the same Basing Street studios" ] }

Creative freedom level: HIGH. The AI invents the show concept. The only constraints are: it must fit the persona's taste territory, it must not repeat a recently-run angle, and the research leads (non-binding suggestions) provide inspiration.

8. Query 6 — Show-Prep Research (Mode B)

📚 Show-Prep Research
brain/llm.py · research_show_prep() → _build_show_prep_prompt() → _query_research() (Mode B)
AI Mode B — Web Search ON

What triggers this

Called occasionally by the show prep pipeline (not every song, not even every show). When a featured artist or theme is being prepared for a deep-dive show, the AI is given WebSearch access and asked to gather real, citable facts. This is the only call with internet access. It's rate-limited and only runs for special programming blocks.

Important: Even in Mode B, ANTHROPIC_API_KEY is always stripped from the environment (unlike Mode A where it depends on BRAIN_LLM_AUTH). This is intentional — the research path uses web session auth, not API key, regardless of the configured auth mode.
SYSTEM PROMPT (SHOW_PREP_PERSONA, line 1295)
You are the research desk for an autonomous freeform radio station, preparing one themed show. Using web search, gather REAL, verifiable facts about the featured artist and theme — genre origins and movements, eras, artist/label/song history, cultural and societal context, the role the music plays in human life. Ground every factual claim in a source you actually found; if you cannot verify a claim, HEDGE it or LEAVE IT OUT — never invent a fact, a date, a quote, or a credit. The recent-output list you are given is ONLY so you AVOID repeating yourself — it is NOT an example to imitate. Respond with ONLY a JSON object {"theme": ..., "tracklist": [{"artist":...,"title":...}], "talking_points": [...]} — no commentary, no markdown fences. 'talking_points' are short framing notes (genre/cultural context), NOT hard factual assertions; the hard facts come from the verified knowledge store, not from you.
USER PROMPT (built by _build_show_prep_prompt(), line 1311)
Prepare a themed show featuring {artist}, theme: {theme}. Use web search to research it. Return ONLY the JSON object described. Already-verified facts (do not re-fetch, build around these): {known_fact_1}; {known_fact_2}; .... Recently aired (AVOID repeating these — this is a repeat-avoidance list, NOT examples to copy): {recent_output_1}; {recent_output_2}; ....
EXPECTED OUTPUT
{ "theme": "Lee 'Scratch' Perry and the Black Ark", "tracklist": [ {"artist": "The Congos", "title": "Heart of the Congos"}, {"artist": "Max Romeo", "title": "War inna Babylon"}, {"artist": "Junior Murvin", "title": "Police and Thieves"} ], "talking_points": [ "The Black Ark studio was built in Perry's garden in Washington Gardens, Kingston, 1973", "Perry burned the studio down in 1983 — accounts differ on whether it was deliberate" ] }

What happens next (Python)

The talking_points go into the show's grounded-facts store (available to future talk prompts for this show). The tracklist suggests tracks for acquisition. The theme informs the show angle. All talking_points are treated as qualified (hedged) facts — never stated as certain on air unless cross-verified with the KNOWLEDGE-008 pipeline.

Creative freedom level: MEDIUM. The AI researches and synthesises. It has web access but must cite sources and hedge anything unverifiable. It cannot invent facts. The Python knowledge pipeline does a final verification pass before any talking_point can be voiced as established fact.

10. Creative Freedom Summary

QueryWhat the AI decides freelyWhat Python constrains/enforces
Track Curation Any artist or song from training knowledge; genre mix; sequencing feel Exclusion lists (already owned, recently played, recently failed). Vetting gate post-download.
DJ Talk Script Phrasing, wit, tone, which facts to highlight, whether to mention the show theme Must use only supplied verified facts. Banned phrases removed. Adversarial check (Query 3) gates delivery.
Adversarial Fact-Check Nothing — analytical role only Output list of claims triggers discard or surgical removal of unsupported claims
Host Identity Name, personality, character voice Gender + age pre-assigned. Taste territory pre-built from real library data. Must output valid JSON.
Show Editorial Angle Theme concept, angle framing, catalog filter phrase, talking point ideas Must fit persona taste. Must not repeat recent angles. Research leads non-binding. Fail-open (returns {} on error).
Show-Prep Research What to search, which sources to trust, how to frame cultural context Must cite/hedge everything. Pre-verified facts supplied (don't re-fetch). KNOWLEDGE-008 does final truth-gate before any claim reaches air.
Bottom line: The AI has genuine creative autonomy over three things — what music to play, what the host says, and what the shows are about. Python handles everything that can't be wrong: scheduling, downloading, fact verification, audio engineering, and the safety gates that prevent hallucinated facts from ever reaching a listener's ears. The AI is the programmer; Python is the studio engineer.
